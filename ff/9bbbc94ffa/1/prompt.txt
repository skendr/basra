# Fix: Game room creation and joining not working

## Context
When clicking "Create Game", no room code is shown and the second player can't join. There are two bugs causing this, plus the server needs to be running.

## Root Cause Analysis

### Bug 1: Socket disconnects between screen navigations (critical)
Every screen calls `useSocket()`, which on mount calls `connectSocket()` and on **unmount** calls `disconnectSocket()` — which destroys the socket instance entirely (`socket.disconnect()` + `socket = null`).

**What happens:**
1. HomeScreen mounts → creates socket A, connects
2. User clicks "Create Game" → emits `create-room` on socket A → navigates to `/lobby`
3. HomeScreen **unmounts** → cleanup runs `disconnectSocket()` → socket A is disconnected and set to null
4. LobbyScreen mounts → `connectSocket()` creates socket B (new connection, new socket ID)
5. Server sends `room-created` response to socket A → dead, never received
6. Server detects socket A disconnect → removes the player/room
7. LobbyScreen shows no room code (roomCode is still null in store)

### Bug 2: Navigation before server response (race condition)
`handleCreate` (index.tsx:45-46) emits `create-room` then immediately calls `router.push('/lobby')` without waiting for the `room-created` response. Same for `handleJoin` (index.tsx:59-60).

### Prerequisite: Server must be running
The server (`npm run server:dev`) is not running — nothing is listening on port 3000.

## Plan

### Step 1: Fix socket lifecycle (`app/src/hooks/useSocket.ts`)
- Remove `disconnectSocket()` from the useEffect cleanup
- Only remove the listeners that this specific hook registered (not `removeAllListeners()`)
- The socket singleton should persist across navigations
- Add a `connect_error` listener so connection failures are visible

### Step 2: Fix navigation timing (`app/app/index.tsx`)
- For "Create Game": listen for `room-created` event and navigate to `/lobby` only after it fires
- For "Join Game": listen for `room-joined` event and navigate to `/lobby` only after it fires
- Show a loading state while waiting for the server response
- Handle errors (show alert if server returns an error)

### Step 3: Fix lobby's duplicate listener issue (`app/app/lobby.tsx`)
- The lobby registers its own `room-joined` listener (line 28-33) to set `opponentName`, but `useSocket` already handles `room-joined` in its effect. For the **joiner**, the lobby's listener should correctly read `opponentNickname`. Verify this works after fixing the socket lifecycle.

## Files to modify
1. `app/src/hooks/useSocket.ts` — fix cleanup, don't disconnect socket
2. `app/app/index.tsx` — wait for server response before navigating
3. `app/src/services/socket.ts` — no changes needed (singleton pattern is fine)
